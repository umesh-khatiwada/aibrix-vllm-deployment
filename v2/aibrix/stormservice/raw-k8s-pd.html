Architecture

This setup deploys SGLang prefill-decode disaggregation across two independent Kubernetes Deployments in the sglang namespace. The prefill server uses 2× AMD MI300X GPUs (TP=2) and runs in --disaggregation-mode prefill, exposing both an HTTP port (8898) and a gRPC bootstrap port (8998) for KV handoff. The decode server also uses 2× MI300X (TP=2) in --disaggregation-mode decode with torch-compile enabled. A mini load balancer coordinates routing between them via the SGLang disaggregation protocol.

Prefill Server

The prefill deployment processes the full input prompt and builds the KV cache using TP=2 across 2× AMD MI300X GPUs. It exposes two ports: the standard HTTP API on 8898 for receiving requests from the load balancer, and a dedicated gRPC bootstrap port on 8998 used by the disaggregation protocol to transfer KV tensors to the decode server. fp8 quantization reduces memory pressure, allowing --mem-fraction-static 0.9.

Launch arguments
ArgumentValue & purpose
--disaggregation-modeprefill — processes prompt, builds KV, never decodes
--disaggregation-bootstrap-port8998 — gRPC port for KV handoff to decode server
--tp-size2 — tensor parallel across 2 MI300X GPUs
--base-gpu-id0 — start GPU assignment from device 0
--quantizationfp8 — 8-bit float reduces VRAM and speeds transfers
--mem-fraction-static0.9 — 90% VRAM reserved for KV cache pool
--disable-radix-cacheDisabled — prevents cache interference in disagg mode
Two ports required: The prefill Service exposes both 8898 (HTTP/named http) and 8998 (gRPC/named grpc-bootstrap). The load balancer hits 8898; the decode server connects to 8998 to pull KV tensors. Both must be reachable via the sglang-prefill ClusterIP Service.

Decode Server

The decode deployment receives KV tensors from the prefill server via the gRPC bootstrap port, then performs pure autoregressive token generation using 2× AMD MI300X GPUs (TP=2). --enable-torch-compile is set for optimised ROCm kernel fusion during generation. --disable-cuda-graph is required because CUDA graphs conflict with the KV injection mechanism used by the disaggregation protocol. Memory fraction is lower (0.7) since the decode side holds received KV — not a full static pool.

Launch arguments
ArgumentValue & purpose
--disaggregation-modedecode — receives KV from prefill, generates tokens only
--disaggregation-bootstrap-port8998 — connects to prefill's gRPC port to pull KV cache
--tp-size2 — tensor parallel across 2 MI300X GPUs
--quantizationfp8 — must match prefill quantization for KV compatibility
--mem-fraction-static0.7 — lower than prefill; decode holds received KV tensors
--enable-torch-compileFuses ROCm kernels — improves decode throughput on MI300X
--disable-cuda-graphRequired — CUDA graphs are incompatible with KV injection
--disable-radix-cacheDisabled — no prefix caching needed in decode-only mode
fp8 must match: Prefill and decode both set --quantization fp8. The KV tensors transferred over gRPC are already in fp8 format — if the decode server uses a different dtype the KV injection will silently corrupt attention outputs.

Mini Load Balancer

The sglang.srt.disaggregation.mini_lb module acts as the disaggregation-aware request coordinator. It receives all client traffic on port 8000 (NodePort 8002) and forwards requests to the prefill and decode servers using their K8s cluster-internal DNS names. It understands the SGLang P/D protocol — it knows to send the prompt to prefill first, wait for KV bootstrap to complete, then forward to decode for generation. Image is v0.4.7 (older stable LB build).

Benchmark Job

A Kubernetes batch Job (bmk-client) runs benchmark_serving.py from the vLLM benchmark suite against the sglang-lb endpoint. It measures TTFT, TPOT, ITL, and E2E latency at the 95th and 99th percentiles. The workload simulates 256 prompts with 1024 input tokens and 128 output tokens at a request rate of 8 req/s with max concurrency of 64. Results are saved to /models/results/sglang_benchmark.json.

Benchmark parameters
FlagValue & purpose
--backendsglang — uses SGLang-compatible request format
--base-urlhttp://sglang-lb:8002 — hits the mini LB NodePort
--request-rate8 req/s — sustained throughput target
--max-concurrency64 — max in-flight requests
--num-prompts256 — total requests in the benchmark run
--random-input-len1024 tokens — simulates medium-length prompts
--random-output-len128 tokens — short generation target
--random-range-ratio0.8 — 80% variance in prompt length
--percentile-metricsttft,tpot,itl,e2el — all key latency dimensions
--metric-percentiles95,99 — tail latency focus
--goodputtpot:25 — SLA: TPOT must be ≤ 25 ms to count as goodput
Results location: /models/results/sglang_benchmark.json on the node's hostPath /home/amd/models. The Job uses backoffLimit: 0 — it will not retry on failure, so check logs immediately if it exits non-zero.
