Architecture Overview

This StormService implements prefill-decode disaggregation across AMD MI300X GPUs. The pipeline splits into three specialized roles: Router orchestrates traffic, Prefill processes prompts and builds KV cache, and Decode (TP=2 across 2 nodes) generates tokens. Mooncake enables high-speed KV cache transfer between phases, eliminating GPU contention.

Router Intelligence

The router acts as traffic director with live Kubernetes service discovery. It maintains endpoints for prefill (role-name=prefill) and decode (role-name=decode) pods on port 30000. Requests are routed using label selectors with --policy random.

Router Launch Command
# sglang-router with PD disaggregation enabled python3 -m sglang_router.launch_router \ --host "0.0.0.0" \ --pd-disaggregation \ --policy "random" \ --service-discovery \ --service-discovery-port "30000" \ --worker-startup-timeout-secs "900" \ --prefill-selector \ storm-service-name=$STORM_SERVICE_NAME,role-name="prefill" \ --decode-selector \ storm-service-name=$STORM_SERVICE_NAME,role-name="decode"

Prefill Processing Layer

A single MI300X GPU handles compute-heavy prompt processing in prefill mode. It generates the KV cache and hands it off to the Mooncake backend over RDMA/Ethernet. Explicit ROCm environment flags are injected to guarantee maximum throughput stability.

Prefill Engine Command
python3 -m sglang.launch_server \ --model-path "/models/Llama-3.2-1B-Instruct" \ --host "0.0.0.0" \ --port "30000" \ --disaggregation-mode "prefill" \ --disaggregation-transfer-backend "mooncake" \ --dtype "bfloat16" \ --tp-size "1" \ --mem-fraction-static "0.8"
⚙️ AMD MI300X ROCm Hardware Optimizations
HSA_NO_SCRATCH_RECLAIM=1Prevents ROCm scratch memory reclamation, stabilizing inference latency.
GPU_FORCE_BLIT_COPY_SIZE=6Optimizes blit copying for small tensor transfers specifically on MI300X architecture.
HIP_FORCE_DEV_KERNARG=1Forces kernel arguments directly into device memory for faster execution.
Distributed Decode Coordination (TP=2)

Memory-bandwidth bound token generation is split across two MI300X GPUs on separate nodes. Rank is derived dynamically from the K8s hostname. A zero-dependency bash script ensures Rank-1 waits for the Rank-0 master port (29500) before initializing NCCL over Ethernet.

Dynamic Rank Derivation & Synchronization Logic
# Extract cluster rank from StatefulSet hostname: <name>-decode-<hash>-<RANK> NODE_RANK=$(echo $HOSTNAME | awk -F'-' '{print $NF}') MASTER_HOSTNAME=$(echo $HOSTNAME | sed 's/-[0-9]*$/-0/') MASTER_DNS="${MASTER_HOSTNAME}.${STORM_SERVICE_NAME}.default.svc.cluster.local:29500" # Rank-1 halts execution and waits for master to open socket if [ "$NODE_RANK" != "0" ]; then until timeout 3 bash -c "echo > /dev/tcp/${MASTER_DNS}" 2>/dev/null; do sleep 3 done fi # Initialize Distributed SGLang python3 -m sglang.launch_server \ --disaggregation-mode "decode" \ --tp-size "2" \ --nnodes "2" \ --node-rank "${NODE_RANK}"
🏎️ Aibrix Gang Scheduling Variables
podGroupSize: 2Ensures Kubernetes schedules both physical nodes atomically (all-or-nothing).
stateful: trueGuarantees predictable DNS hostnames required for the bash rank derivation logic.
NCCL_IB_DISABLE=1Forces standard Ethernet usage by actively disabling InfiniBand protocols.
