Architecture

This StormService achieves prefill-decode disaggregation using vLLM as the inference engine and LMCache as the KV transfer layer. The prefill pod runs vLLM in kv_producer mode — it processes the prompt, builds the full KV cache, and pushes it to the LMCache server. The decode pod runs vLLM in kv_consumer mode — it pulls that cached KV and streams tokens without ever re-running prefill. The router directs requests to the correct role via Kubernetes service discovery.

Routing Layer

The router runs on CPU only — no GPU required. It uses Kubernetes service discovery to maintain a live map of all prefill and decode pods. It selects worker pods by matching three labels: storm-service-name, role-name, and role-replica-index. This means it can precisely target a specific replica within a specific role regardless of pod IP changes.

Prefill Engine

The prefill pod runs vLLM in kv_producer mode. It ingests the full prompt, runs the model forward pass to build the attention KV cache, and immediately pushes those KV tensors to the LMCache server via gRPC. It never generates a single output token — its only responsibility is prompt processing and KV population. The LMCACHE_USE_EXPERIMENTAL=True env var is mandatory or the connector is silently skipped.

vLLM KV transfer config — prefill
ParameterValue & purpose
kv_connectorLMCacheConnectorV2 — plugs into vLLM's attention transfer hook
kv_rolekv_producer — builds KV cache and pushes to LMCache
lmcache_rpc_portlmcache-server:8400 — gRPC address of the LMCache K8s service
kv_buffer_devicecuda — KV stays on GPU, avoids PCIe round-trip to CPU
Required env vars
VariableEffect
LMCACHE_USE_EXPERIMENTALTrue — activates LMCacheConnectorV2 plugin in vLLM

Model path: Loaded from hostPath /data01/models → /models (readOnly). Model must be pre-staged on every schedulable node. Use a PVC in production to avoid silent failures on rescheduling.
container: prefill · vllm launch
# Image: vllm/vllm-openai:latest # Resources: nvidia.com/gpu: 1 # Env: LMCACHE_USE_EXPERIMENTAL=True python3 -m vllm.entrypoints.openai.api_server \ --model /models/Llama-3.1-8B-Instruct \ --served-model-name Llama-3.1-8B-Instruct \ --host 0.0.0.0 \ --port 30000 \ --gpu-memory-utilization 0.8 \ --dtype bfloat16 \ --max-model-len 8192 \ --trust-remote-code \ --kv-transfer-config '{ "kv_connector": "LMCacheConnectorV2", "kv_role": "kv_producer", "kv_connector_extra_config": { "lmcache_rpc_port": "lmcache-server:8400", "kv_buffer_device": "cuda" } }' # Volumes: # model-vol → hostPath /data01/models (readOnly) # shared-mem → emptyDir Memory (/dev/shm)
Decode Engine

The decode pod runs vLLM in kv_consumer mode. On receiving a request, it queries LMCache for a pre-built KV matching the prompt hash. If found, it loads those tensors directly into GPU memory and begins autoregressive token generation — the prefill computation is entirely skipped. This dramatically reduces time-to-first-token and frees the decode GPU for pure generation throughput.

vLLM KV transfer config — decode
ParameterValue & purpose
kv_connectorLMCacheConnectorV2 — same plugin, consumer side
kv_rolekv_consumer — pulls KV from LMCache, skips prefill pass
lmcache_rpc_portlmcache-server:8400 — same LMCache K8s service endpoint
kv_buffer_devicecuda — KV loaded directly into decode GPU memory
Startup dependency
OrderService & condition
① Firstlmcache-server pod must be Ready — decode init will fail otherwise
② Secondprefill pod up — router won't route until both workers are reachable
③ Thirddecode pod starts — connects to LMCache on init, then signals Ready
KV cache hit guarantee: Prefill and decode must receive the identical tokenised prompt. Any mismatch in text or tokeniser config causes a cache miss — the decode node falls back to running prefill itself, negating the disaggregation benefit entirely.

StormService CRD

The StormService custom resource packages the entire vLLM P/D stack — router, prefill, decode, and LMCache server — into a single deployable object. Setting stateful: true at both service and role level guarantees stable pod DNS names needed for reliable service discovery. The LMCache server role carries no GPU resource request, keeping it on CPU nodes only.
